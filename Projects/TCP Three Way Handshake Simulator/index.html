TCP Connection Management Details

Three-Way Handshake (Establishment)

To establish a reliable connection over IP, TCP uses a three-way handshake to synchronize Sequence Numbers and allocate buffer resources:

  • 1. SYN: The Client sends a segment with flag SYN=1, ACK=0 and its Initial Sequence Number ($Seq = X$). Sockets moves to SYN-SENT.
  • 2. SYN-ACK: The Server replies with SYN=1, ACK=1, its own Initial Sequence Number ($Seq = Y$), and sets $Ack = X + 1$ to acknowledge the Client's SYN segment. State becomes SYN-RECEIVED.
  • 3. ACK: The Client replies with SYN=0, ACK=1, setting $Seq = X + 1$ and $Ack = Y + 1$ to acknowledge the Server's SYN-ACK. Sockets transition to ESTABLISHED on both sides.

Four-Way Handshake (Teardown)

Because TCP connections are full-duplex (allowing data to flow in both directions independently), closing a connection requires a four-way handshake to shut down both halves:

  • 1. FIN: Active close node sends FIN=1 (e.g. $Seq = U$), moving to FIN-WAIT-1 state.
  • 2. ACK: Receiver replies with ACK=1 ($Ack = U + 1$), transitioning to CLOSE-WAIT. Active close node enters FIN-WAIT-2.
  • 3. FIN: Once the receiver completes its own final data writes, it sends its own FIN=1 (e.g. $Seq = W$), moving to LAST-ACK.
  • 4. ACK: Active close node replies with ACK=1 ($Ack = W + 1$) and enters TIME-WAIT for a duration of $2 \times \text{MSL}$ (Maximum Segment Lifetime) before closing completely.

Timers & Retransmission (RTO)

To guarantee delivery over unreliable links, TCP utilizes timers and acknowledgment validation:

  • Sequence Bookkeeping: In TCP, sequence numbers count *data bytes*, not packets. However, control flags (like SYN and FIN) consume exactly 1 sequence number logically. Acknowledge segment numbers always declare the *next byte expected*.
  • Retransmission Timeout (RTO): When a node sends a segment, it launches a timer. If it does not receive an acknowledgment before the timer reaches 0, it assumes packet loss and automatically retransmits the segment.
